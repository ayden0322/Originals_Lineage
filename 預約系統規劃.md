## 一、系統概述與核心變更

### 原設計（已實作的事前預約功能 - Commit 2405569）
- email / phone / lineId 記錄
- Email 驗證機制
- 里程碑系統
- 管理後台設計頁面

### 新規劃（完整重構方案）
- **預約必須登入遊戲帳號**（而非匿名 email 預約）
- 預約人數里程碑獎勵機制（手動設置多個獎勵階段）
- 進度條展示
- 結算日期與發放流程
- 後台管理完整配置

### 關鍵決定點
1. 預約必須登入綁定遊戲帳號（完全去掉 email 預約）
2. 未登入點擊預約頁 → 導向登入頁並帶 `redirect` 參數，登入完自動跳回
3. 多階層里程碑獎勵（手動設置）
4. 種子數字 + 實際預約數 = 顯示人數（僅用於顯示 / 達標判斷）
5. **發放對象僅限實際預約名單**（種子人數不發獎）
6. 結算日期後統一發放（非實時發放）
7. **預約後不可取消**（一人一次，預約完成即鎖定）
8. 後台完整配置（文字、圖片、獎勵、發放 log）

### 兩個人數概念（工程師必讀）

| 概念 | 計算 | 用途 |
|---|---|---|
| 顯示人數 | `count_base（種子）+ 實際預約數` | 前台顯示、里程碑達標判斷 |
| 實際名單 | 只有 `reservations` 表的真實資料 | 發獎勵的對象 |

**例**：種子 400 + 真實 100 = 顯示 500 → 500 人里程碑達標 → 但只發獎給那 100 個真實預約者。

## 二、功能需求（用戶視角）

### 前台預約頁面流程
```
未登入 → 導向 /auth/login?redirect=/public/reserve
            ↓（登入成功自動跳回）
玩家登入（需綁定遊戲帳號）→ 進入預約頁
  ↓
已預約？ → 顯示「已完成事前預約」（無取消選項）
未預約？ → 勾選同意 + 按「立即預約」→ 綁定當前登入帳號
```

### 人數與里程碑顯示
- **顯示人數計算**：`count_base（手動種子數字）+ COUNT(已預約人數)`
- **里程碑達標判斷**：`顯示人數 >= 里程碑閾值`
- **多層級獎勵**：支援自訂階段數（例如 300 人、500 人、800 人等多個里程碑）
- **達標時發放所有達標獎勵**：如達 500 人，則自動發放 300 人 + 500 人獎勵

### 結算與暫停
- **結算日期**：後台設定截止日期（`deadline_at`）
- 到期後自動停止預約申請
- 結算後鎖定（`is_distribution_locked`）防止意外操作
- **管理者可提前發放**：只要手動鎖定 `is_distribution_locked = true`，即可在 deadline 前執行發放（例如活動提前結束）

## 三、資料結構 / 資料庫設計

### 3-1. `reservations` 表（完全重構）

```sql
id                uuid PK
website_user_id   uuid FK → accounts（登入時綁定）
game_account_name varchar（快照：預約當下對應的遊戲帳號）
ip_address        varchar（稽核用，不做防刷）
created_at        timestamp

UNIQUE (website_user_id)  -- 一人一次，預約後不可取消
```

**移除欄位**：`email` / `displayName` / `phone` / `lineId` / `emailVerified` / `verificationCode` / `status` / `updated_at`

**設計簡化說明**：
- 因為不開放取消，沒有 `status` 狀態機的必要，也不需要 `updated_at`
- 一旦寫入即永久有效，`UNIQUE (website_user_id)` 自然保證一人一次

### 3-2. `reservation_milestones` 表（擴充）

```sql
id                  uuid PK
threshold           int（人數門檻，如 300）
reward_name         varchar（簡短名稱，如「限定坐騎」）
reward_description  text（富文本詳細說明）
image_url           varchar（獎勵圖 MinIO URL）
sort_order          int
is_active           boolean
created_at          timestamp
updated_at          timestamp
```

### 3-3. `reservation_page_settings` 表（新建，單筆設定）

```sql
id                      uuid PK（固定單筆）
page_title              varchar（如「事前預約活動」）
page_subtitle           varchar
page_description        text（富文本，頁面主說明文）
count_base              int（手動種子數字，預設 0）
deadline_at             timestamp（結算日，到期後停止預約）
is_distribution_locked  boolean（結算後鎖定，防止意外操作）
updated_at              timestamp
```

### 3-4. `reservation_reward_batches` 表（發放批次記錄）

```sql
id                  uuid PK
triggered_by        uuid（操作員）
total_recipients    int（應發人數）
delivered_count     int
failed_count        int
status              enum: 'running' / 'completed' / 'failed'
started_at          timestamp
completed_at        timestamp
error_summary       text
```

### 3-4b. `reservation_reward_batch_milestones` 表（批次 × 里程碑 join table）

```sql
batch_id        uuid FK → reservation_reward_batches
milestone_id    uuid FK → reservation_milestones

PRIMARY KEY (batch_id, milestone_id)
```

**理由**：用 join table 而非 `uuid[]` 陣列，好處是可以反查「某里程碑的發放歷史」、支援 foreign key 完整性、SQL 寫法標準。

### 3-5. `reservation_reward_claims` 表（個別發放紀錄）

```sql
id                  uuid PK
batch_id            uuid FK → reservation_reward_batches
reservation_id      uuid FK → reservations
milestone_id        uuid FK → reservation_milestones
game_account_name   varchar（發放時快照）
status              enum: 'delivered' / 'failed' / 'skipped'
delivered_at        timestamp
error_message       text

UNIQUE (batch_id, reservation_id, milestone_id)  -- 防重
```

### 3-6. `site_settings` 新增欄位

```sql
post_register_redirect_url  varchar
  -- 註冊完成後跳轉頁面，預設 /public/reserve
  -- 允許空值 → fallback 到預設首頁
  -- 例如：/public/reserve（事前預約）、/public/newbie（新手福利）、/public（首頁）
```

## 五、流程設計

### 5-1. 預約流程

```
未登入 → 導向 /auth/login?redirect=/public/reserve
  ↓
玩家登入 → 查詢預約狀態（已預約?）
  ↓
展示：
  - 當前人數 + 進度條
  - 各階段獎勵圖片 + 描述
  - 倒數時間 / 結算日期
  ↓
已預約 → 顯示「已完成事前預約」（無按鈕）
未預約 → 勾選同意 + 立即預約
  ↓
檢查：now < deadline_at 且未鎖定？
      MemberBinding 綁定遊戲帳號？
  ↓
寫入 reservations
回傳成功 → 頁面更新顯示「已完成事前預約」
```

### 5-2. 達標判斷邏輯

```
實時計算：
  displayCount = count_base + COUNT(reservations)
  （沒有狀態欄位，每一筆都有效）

達標列表：
  SELECT * FROM reservation_milestones
  WHERE threshold <= displayCount AND is_active = true
  ORDER BY threshold ASC

發放對象（重要）：
  實際名單 = SELECT * FROM reservations  -- 不含種子人數
```

### 5-3. 發放流程（後台操作）

**前置條件**（擇一即可）：
- `now >= deadline_at`（結算日已到）
- 或管理者手動將 `is_distribution_locked` 設為 true（提前結束活動）

且目前無其他 `running` 狀態的批次在執行中

**發放步驟**：
```
管理者點「執行發放」按鈕
  ↓
1. 鎖定 is_distribution_locked = true
2. 建立 reservation_reward_batches 紀錄
3. 對每個「已預約」× 每個「已達標 milestone」建 reservation_reward_claims 紀錄
4. 非同步執行：呼叫 gameDbService.insertGiftReward()
5. 更新 batch 進度與 claim 狀態
6. 寫入白話 log → 後台可查看
  ↓
完成 / 失敗 claim 可單筆重試
```

### 5-4. 發放 Log 白話紀錄範例

```
[2026-04-20 14:30] 開始發放：達標里程碑 2 個（300人、500人）
[2026-04-20 14:30] 應發放對象：450 人
[2026-04-20 14:31] 已發放 100/450 — 成功 98、失敗 2
[2026-04-20 14:32] 玩家「lineage001」發放失敗：遊戲帳號不存在
[2026-04-20 14:35] 發放完成：成功 445、失敗 5
```

## 九、核心邏輯細節

### 9-1. 非同步發放
- 批次發放可能涉及幾百筆記錄
- 使用非同步隊列或簡單 async loop + DB transaction 分批 commit
- 實時更新 batch 進度

### 9-2. 冪等性保護
- `reservation_reward_claims` 的唯一索引 `(batch_id, reservation_id, milestone_id)` 擋重複發放

### 9-3. 失敗重試機制
- 參考現有 `shop.service.ts` retry 邏輯
- **僅對連線類錯誤重試**（timeout / connection refused / 網路中斷）
- **資料類錯誤不重試**（遊戲帳號不存在、SQL 語法錯誤、欄位錯誤）→ 直接標 `failed`
- 重試間隔採指數退避：30s → 2min → 10min，共 3 次
- 重試全部失敗後更新 `reservation_reward_claims.status = 'failed'`，於後台批次詳情中顯示「失敗原因」與「重試」按鈕，供管理者手動處理

### 9-4. 寶物來源
- 共用商城 Product 表（`gameItemId + gameItemName`）
- 或純手動輸入 ID
- 建議支援兩者

## 十、風險與注意事項

| 風險 | 對策 |
|------|------|
| 重複發放 | DB 唯一索引 `(batch_id, reservation_id, milestone_id)` + 批次狀態雙重保護 |
| 批次太大超時 | 非同步處理，進度即時更新 |
| 遊戲 DB 連線失敗 | 連線類錯誤進指數退避重試；3 次失敗後標 failed，後台可手動重試 |
| 遊戲帳號不存在（資料錯誤）| 直接標 failed，後台顯示失敗原因交由管理者處理 |
| 結算後誤操作 | `is_distribution_locked` 鎖定 |
| 未登入玩家誤點預約 | 前端導向登入頁並帶 `redirect`，登入後自動回預約頁 |

註：因為預約必須登入且綁定遊戲帳號，舊方案「預約人未註冊」的風險已不存在，相關補發 Hook 也移除。
