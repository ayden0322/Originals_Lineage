## 一、系統概述

### 1.1 系統目的

為始祖天堂（Originals Lineage）建立一個二層代理分潤系統。玩家透過代理的推廣連結進站註冊並儲值後，系統自動依照「上游切下來」的方式把儲值金額的一部分分給對應代理作為分潤。

### 1.2 核心特性

- **兩層代理結構**（A → B），資料表保留 `parent_id` 欄位，未來擴多層不用改 schema
- **分潤基準 = 玩家儲值金額**（不扣毛利、不扣手續費）
- **「上游切下來」計算方式**：管理者給 A 30%，A 給 B 的 % 是從「A 拿到的 30%」裡面再切
- **歷史快照原則**：所有比例、歸屬的變更都以時間快照方式存檔，歷史分潤紀錄永不重算
- **動態結算週期**：管理者設定每月幾號結算，系統自動按該日切期
- **管理者最大權限**：可調整玩家歸屬、調代理父級、停權代理、手動加減項

### 1.3 技術棧對應

- 後端：NestJS + TypeORM，所屬模組 `modules/originals-lineage/commission`
- 資料庫：PostgreSQL（主庫）
- 排程：`@nestjs/schedule`（結算日檢查 cron）

## 二、角色與權限

### 2.1 三種角色

| 角色 | 說明 | 登入位置 |
|------|------|----------|
| **管理者** | 始祖天堂營運方，擁有全系統最高權限 | 總後台 `/admin` |
| **A 代理（一級代理）** | 管理者直接授權開通 | 代理後台 |
| **B 代理（二級代理）** | 由管理者建立、掛在某個 A 底下 | 代理後台 |
| **玩家** | 站內會員 | 前台網站 |

### 2.2 系統虛擬代理

另有一個特殊的 **SYSTEM 虛擬代理**：
- 用來承接「無歸屬玩家」的所有儲值紀錄
- `agent_id = SYSTEM`（可用固定 UUID 或 `code = 'SYSTEM'` 辨識）
- 所有這類玩家的儲值都會產生一筆 `commission_records`（`amount = 0`），純粹用來報表加總閉環
- 不會實際分潤給任何人

### 2.3 權限能做什麼

| 動作 | 管理者 | A 代理 | B 代理 |
|------|:------:|:------:|:------:|
| 新增 A 代理 | ✅ | ❌ | ❌ |
| 新增 B 代理 | ✅ | ❌（需線下溝通） | ❌ |
| 設定 A 的分潤 % | ✅ | ❌ | ❌ |
| 設定 B 的分潤 %（初始值） | ✅ | ❌ | ❌ |
| 設定 B 的分潤 %（後續調整） | ✅ | ✅（需管理者開啟 `can_set_sub_rate`） | ❌ |
| 停權 / 啟用代理 | ✅ | ❌ | ❌ |
| 調整玩家歸屬 | ✅ | ❌ | ❌ |
| 調整代理父級（B 轉掛到別的 A） | ✅ | ❌ | ❌ |
| 產生 / 停用推廣連結 | ✅ | ✅（自己的） | ✅（自己的） |
| 查看自己的業績報表 | ✅（全部） | ✅（自己 + 旗下 B） | ✅（自己） |
| 查看玩家消費明細 | ✅（完整） | ✅（遮罩） | ✅（遮罩） |
| 手動退款沖銷 | ✅ | ❌ | ❌ |
| 結算確認 | ✅ | ❌ | ❌ |

## 三、代理建立與設定

### 3.1 新增 A 代理（管理者操作）

管理者在總後台點「新增代理」，填入：
- 代理名稱
- 登入帳號
- 登入密碼
- 聯絡資訊（選填）
- 分潤比例（例如 30%）
- `self_referral_allowed` 開關（是否允許此 A 自推自玩）
- `can_set_sub_rate` 開關（是否允許此 A 自行調整旗下 B 的分潤比例）

系統執行：
1. 寫入 `agents` 表
2. 寫入 `agent_rates` 表一筆初始比例（`effective_from = now()`, `effective_to = NULL`）
3. 自動產生一組預設推廣連結 + QR Code，寫入 `referral_links`

### 3.2 新增 B 代理（管理者操作）

B **一律由管理者統一建立**，A 不能自己新增下線（若 A 要擴充團隊，需線下溝通請管理者建立）。

管理者在總後台點「新增子代理」時選擇所屬 A，填入：
- B 的基本資料（名稱、帳號、密碼、聯絡資訊）
- B 的分潤比例**初始值**（例如 60%）
- B 掛在哪個 A 底下（`parent_id`）

### 3.3 A 自行調整 B 的比例（視開關而定）

- **若 `A.can_set_sub_rate = false`**：A 的後台看不到「調整 B 比例」的 UI，只能查看 B 當前的比例
- **若 `A.can_set_sub_rate = true`**：A 的後台可以修改旗下 B 的比例，範圍 0~100%（無上限）
- 無論哪種情況，管理者都能改

### 3.4 比例變更的生效邏輯

- **即時生效**（不需等下期）
- 變更流程：把原本那筆 `agent_rates.effective_to` 從 `NULL` 改成 `now()`，再新增一筆 `effective_from = now(), effective_to = NULL` 的新紀錄
- 歷史已產生的 `commission_records` 完全不動（快照已存 `rate_snapshot`）

## 四、分潤計算規則

### 4.1 計算公式

**基準**：玩家每次儲值金額（交易成功 `status = 'paid'` 時觸發）

**情境一：玩家歸屬在 B（二級代理）**

```
團隊池 = 儲值金額 × A.rate
B 實拿 = 團隊池 × B.rate
A 實拿 = 團隊池 − B 實拿
```

**範例**：玩家儲值 100 元，A 比例 30%、B 比例 60%
- 團隊池 = 100 × 30% = **30 元**
- B 實拿 = 30 × 60% = **18 元**
- A 實拿 = 30 − 18 = **12 元**

**情境二：玩家歸屬在 A（一級代理）**

```
A 實拿 = 儲值金額 × A.rate
```

**範例**：玩家儲值 100 元，A 比例 30%
- A 實拿 = 100 × 30% = **30 元**（全拿，不分給 B）

**情境三：玩家無歸屬（歸屬 SYSTEM）**

- 不實際分潤
- 仍寫入一筆 `commission_records`（`agent_id = SYSTEM, rate = 0, amount = 0`），保證報表加總閉環

### 4.2 觸發時機

- 金流 callback 將 `transactions.status` 設為 `'paid'` 的瞬間
- 呼叫 `CommissionService.generateCommissions(tx)`，即時展開成 `commission_records`
- **不要等結算時才算**（資料量大會爆）

### 4.3 比例快照

每筆 `commission_records` 都要存：
- `rate_snapshot`：該代理在交易時刻的有效比例
- `upstream_rate_snapshot`：上游代理的比例（算 A 殘留用，B 才需要，A 則為 NULL）

這樣未來 rate 變更 100 次，歷史分潤紀錄永遠不變。

## 五、玩家歸屬規則

### 5.1 歸屬來源

| 來源 | `linked_source` | 說明 |
|------|----------------|------|
| Cookie | `cookie` | 玩家掃 QR / 點連結時寫 Cookie，註冊時讀取 |
| 註冊時直接帶參數 | `register` | URL 有 `?ref=xxx` 直接註冊沒經過 Cookie |
| 手動指定 | `manual` | 管理者後台手動綁定 |
| 無歸屬 | `system` | 沒 Cookie、沒 ref 參數，歸 SYSTEM |

### 5.2 Cookie 設定

- 名稱：`ref_code`
- 有效期：**30 天**（可在系統設定調整）
- 覆蓋規則：**後者覆蓋**。玩家註冊前，掃多個 QR，以最後一個為準
- 註冊完成後，Cookie 失效與否不影響歸屬（已寫入 `player_attribution`）

### 5.3 歸屬寫入時機

- 玩家「註冊成功」當下，後端讀 Cookie → 查 `referral_links` → 找到 agent → 寫入 `player_attribution`
- 寫入後 **終身綁定**，玩家之後掃別人的 QR 一律不影響

### 5.4 自推自玩偵測

- 僅當 `agent.self_referral_allowed = false` 時才檢查
- 檢查條件（擇一或全部命中即視為自推，阻擋註冊或不綁歸屬，由管理者設定）：
  - 註冊 IP 與代理登入 IP 相同
  - 註冊手機/Email 與代理資料相同
  - 同一裝置指紋

### 5.5 歸屬調整（管理者操作）

- 總後台「玩家歸屬管理」頁面可直接改 `player_attribution.agent_id`
- 每次改動寫一筆 `player_attribution_history`（`from_agent_id, to_agent_id, changed_by, changed_at, reason`）
- **歷史 `commission_records` 不動**，只影響調整之後的新交易
- 管理者可對 Cookie 有歸屬的玩家強制改成 SYSTEM 或其他代理

## 六、結算週期規則

### 6.1 結算日設定

- 管理者在系統設定設定 `settlement_day`（1~31 的整數）
- 系統排程每天 00:00 檢查：今天是不是結算日？是的話執行結算

### 6.2 月底邊界處理

- 若 `settlement_day > 當月天數`（例如設 31 號但 2 月只有 28 天）→ **抓當月最後一天結算**
- 範例：設 31 號，2 月在 2/28 結算，4 月在 4/30 結算，5 月在 5/31 結算

### 6.3 時間切點

- 採用**左閉右開區間** `[start, end)`
- 範例：結算日 5 號 → 4 月這一期是 `[2026-04-05 00:00:00, 2026-05-05 00:00:00)`
- 5/5 00:00:00 的交易算下一期

### 6.4 結算流程

系統每天 00:00 跑 cron：

```
if today.day == settlement_day (or today == 當月最後一天 且 settlement_day > 當月天數):
  1. 計算上一期範圍 [prev_start, today_00:00)
  2. 撈出該範圍內所有 commission_records WHERE settlement_id IS NULL
  3. 按 agent_id 分組加總
  4. 為每個代理建立一筆 settlements（status = 'pending'）
  5. 把對應 commission_records.settlement_id 填入
```

### 6.5 結算確認

管理者進「結算管理」頁：
- 看到每個代理本期的 `total_commission`
- 可新增 `settlement_adjustments`（加減項，例如退款沖銷）
- 系統重算 `final_amount = total_commission + sum(adjustments)`
- 點「確認結算」→ `status = 'settled'`, `settled_at`, `settled_by`

**結算後的狀態流**：
```
pending（計算完待確認）
  → settled（管理者確認）
  → paid（線下轉帳完成，管理者手動標記）
```

### 6.6 結算日中途變更

採用**預約下期生效**策略：

- 管理者在系統設定改結算日時，系統跳提示：「此變更將從下一期結算後套用，當期仍依原結算日 X 號切結。」
- 變更寫進 `system_settings_history`（`key, new_value, effective_from_period, changed_by`）
- 每次 cron 執行時先撈「下一期該用哪個結算日」：
  - 若變更的 `effective_from_period` 是下一期 → 用新值
  - 否則用舊值

**過渡期長度不一**（例如舊 5 號 → 新 10 號，過渡期變 36 天）是**預期行為**，不追溯、不切割，報表如實呈現。

**首次設定**：第一期 = `[系統啟用日, 第一個結算日 00:00)`，可能只有幾天，接受它。

## 七、退款與訂單沖銷

### 7.1 訂單狀態設計

`transactions.status` 可能值：
- `pending`：金流處理中
- `paid`：交易完成（**觸發分潤計算**）
- `failed`：交易失敗
- `refunded`：完成交易（手動退款）

### 7.2 退款處理流程

1. 玩家申請退款 → 客服處理 → 實際退款（線下）
2. 管理者在訂單管理頁把該訂單改為 `refunded`
3. 系統自動執行「一鍵沖銷」：
   - 找出該 `transaction_id` 對應的所有 `commission_records`
   - 為每一筆寫入對應的 `settlement_adjustments`（負值）
     - 若該 commission 尚未結算（`settlement_id IS NULL`）：等該期結算時自動扣
     - 若該 commission 已結算（`settlement_id` 有值）：加到**當期新的 settlement** 作為負項（跨期沖銷）
4. 業績報表自動顯示扣項

### 7.3 沖銷範例

玩家儲值 100（已產生：A 拿 12、B 拿 18）後退款：
- 系統自動產生兩筆調整：
  - `{agent: A, amount: -12, reason: '玩家XX退款，訂單 #123'}`
  - `{agent: B, amount: -18, reason: '玩家XX退款，訂單 #123'}`

## 八、管理者介入情境

### 8.1 停權代理

管理者把 `agents.status` 改成 `suspended`：
- **若 A 底下仍有 B**：系統跳提示「代理 A 底下還有子代理 B1, B2...，請先轉移再停權」，**不允許直接停權**
- A 停權後，A 的推廣連結全部失效（新玩家掃 QR 不再綁定 A）
- A 的代理後台禁止登入
- A 旗下玩家的**新交易不再產生 A/B 的分潤**（進 SYSTEM）
- 歷史分潤仍可照常結算、確認、領完

### 8.2 B 轉掛到別的 A

管理者操作「B 轉組」：
- `agents.parent_id` 從 A 改成 A2
- **歷史 `commission_records` 不動**（A 過去帶 B 的抽成保留）
- 新交易進來時，B 的 parent 已是 A2 → 分潤給 A2
- 建議寫一筆變更軌跡（`agent_parent_history` 表，選配）

### 8.3 玩家歸屬調整

見 [5.5](#55-歸屬調整管理者操作)

### 8.4 手動加減項

管理者在結算頁可對任何代理、任何期新增 `settlement_adjustments`：
- 加項：補發獎金、回饋、調整
- 減項：退款沖銷、錯誤扣回
- 每筆都要填 `reason`，留紀錄

### 8.5 B 升格為一級代理（A）

**情境**：A 離職、B 業績獨立、客戶分家等需要把 B 拉到頂層的場合。

**規則**：
- 該代理必須是 B（有 `parentId`）且 `status = 'active'`
- 必須提供新 rate（原 60% 是「從 A 拿到的 30 元中切 60%」，升格後變成「直接拿玩家儲值的 X%」，比例必須重設）
- 必須填寫升格原因（稽核用）
- **歷史完全不動**：旗下玩家歸屬保留、歷史 `commission_records` 保留、過去結算保留
- **rate 採時段快照**：舊 rate 寫 `effective_to = now()`，新 rate 寫一筆 `effective_from = now()`
- 寫一筆 `commission_agent_parent_history`（`action = 'promote'`，含舊/新 rate 快照與原因）
- 升格後完全套用「A 代理」既有規則：
  - 旗下可由管理者新增子代理（B'）
  - B' 的初始 rate 由管理者建立時設定
  - 是否允許自設 B' 的 rate，由 `can_set_sub_rate` 開關控制（既有欄位）
- **計算範例**：升格後 rate = 18%，管理者新建 B' 掛在它底下且 B' = 50%
  - 玩家儲值 100 元 → 團隊池 18 元 → B' 拿 9 元 → 升格後的 A 留 9 元

**API**：
```
POST /modules/originals/commission/agents/:id/promote
Body: { newRate: 0.18, reason: "A001 離職，獨立經營" }
```

**前端 UX 建議**（兩步驟 Modal）：
1. 步驟一：警示說明會發生什麼、保留什麼，強調可逆
2. 步驟二：設定新 rate（預設帶入「等價值」`oldA × oldB`）+ 即時試算 + 強制原因

### 8.6 B 轉組（換掛到別的 A）

- `agents.parent_id` 從 A 改成 A2
- 寫一筆 `commission_agent_parent_history`（`action = 'change_parent'`）
- **歷史 `commission_records` 不動**（A 過去帶 B 的抽成保留）
- 新交易進來時，B 的 parent 已是 A2 → 分潤給 A2

## 九、報表與隱私規則

### 9.1 代理後台（A 看到的）

- **本期預估分潤**：`SUM(commission_records WHERE agent_id = A AND settlement_id IS NULL)`
- **本期團隊業績**：旗下所有玩家（含 A 直推 + B 旗下）的儲值加總
- **子代理業績明細**：列出 B1、B2...
  - 顯示：B 帶來的總業績、B 自己分走多少
  - **不顯示**：A 從 B 身上抽了多少（避免 B 看到時有情緒）
- **玩家消費明細**：玩家 ID（遮罩）、消費金額、消費日期
- **歷史結算列表**：每期結算狀態 + CSV 下載

### 9.2 代理後台（B 看到的）

- 本期預估分潤
- 自己帶來的業績總額、自己分到多少
- 自己旗下玩家的消費明細（玩家 ID 遮罩）
- **看不到**：A 的分潤細節、同級 B 的業績

### 9.3 管理者後台

- 全代理樹狀結構
- 排行榜（業績、分潤金額、活躍玩家數）
- 異常偵測（退款率、自推疑似、業績突增）
- 任何代理的完整明細（玩家 ID 可選擇不遮罩）
- 全平台 CSV 匯出

### 9.4 玩家 ID 遮罩規則

- 預設格式：`player_***001`（保留末三碼）
- `system_settings.mask_player_id_for_agents = true/false`
- 管理者可全域開關是否對代理揭露完整玩家 ID

### 9.5 CSV 匯出欄位

```
交易ID, 玩家ID（視遮罩設定）, 儲值金額, 儲值時間, 你的分潤比例, 分潤金額, 歸屬層級, 結算期
```

## 十二、邊界案例彙總

### 12.1 時間邊界

| 情境 | 處理 |
|------|------|
| 結算日 = 31，2 月 | 抓 2/28（或 2/29） |
| 交易時間落在結算日 00:00:00 整 | 算下一期（`[start, end)` 左閉右開） |
| 中途改結算日 | 下一期才生效，當期照舊 |
| 首次設結算日 | 第一期從系統啟用日起算到首個結算日 |

### 12.2 歸屬邊界

| 情境 | 處理 |
|------|------|
| 無 Cookie 註冊 | 歸 SYSTEM |
| Cookie 過期（>30 天） | 視為無歸屬，歸 SYSTEM |
| 多次掃不同 QR | 後者覆蓋，以註冊前最後一個為準 |
| 註冊後再掃新 QR | 不影響歸屬（已終身綁定） |
| 代理 A 被停權後，A 旗下玩家儲值 | 進 SYSTEM（A 旗下 B 也連帶停用） |
| 推廣連結停用後玩家掃 QR | 視為無歸屬 |

### 12.3 比例邊界

| 情境 | 處理 |
|------|------|
| A 設定 B 比例 = 100% | 允許（若 `can_set_sub_rate = true`），A 自己拿 0 |
| A 設定 B 比例 = 0% | 允許，B 拿 0，A 全拿 |
| 比例變更瞬間的交易 | 用 `paid_at` 去查 `agent_rates`，剛好在邊界的以 `effective_from <= paid_at` 為準 |

### 12.4 退款邊界

| 情境 | 處理 |
|------|------|
| 退款時該筆分潤未結算 | 寫入對應期的 settlement_adjustments，等該期結算時自動扣 |
| 退款時該筆分潤已結算（已 settled 但未 paid） | 加到當期新的 settlement 作為負項（跨期沖銷） |
| 退款時該筆分潤已出款 | 同上，下期才扣（從代理當期收益扣掉） |
| 退款造成該代理當期分潤為負 | 允許負數結算，下期滾入繼續扣直到清零 |

### 12.5 代理生命週期邊界

| 情境 | 處理 |
|------|------|
| 停權 A 時 A 還有 B | 擋下來提示先轉移 B |
| 停權 B（底下沒人） | 直接停權，B 的推廣連結失效 |
| 恢復已停權的代理 | 可恢復，但其原推廣連結默認繼續失效（管理者手動啟用） |

### 分潤計算引擎（TypeScript 虛擬碼）

```typescript
async function generateCommissions(tx: Transaction) {
  const attr = await getAttribution(tx.playerId);

  // 無歸屬 → SYSTEM
  if (!attr || attr.agentId === SYSTEM_AGENT_ID) {
    await writeCommissionRecord({
      agent: SYSTEM_AGENT_ID,
      level: 1,
      base: tx.amount,
      rate: 0,
      amount: 0,
    });
    return;
  }

  const owner = await getAgent(attr.agentId);

  // 檢查停權
  if (owner.status === 'suspended' ||
      (owner.parent_id && (await getAgent(owner.parent_id)).status === 'suspended')) {
    await writeCommissionRecord({ agent: SYSTEM_AGENT_ID, ... });
    return;
  }

  if (owner.parent_id) {
    // Case 1: 玩家歸屬在 B
    const A = await getAgent(owner.parent_id);
    const rateA = await getEffectiveRate(A.id, tx.paidAt);
    const rateB = await getEffectiveRate(owner.id, tx.paidAt);

    const teamPool = tx.amount * rateA;
    const bCut = teamPool * rateB;
    const aCut = teamPool - bCut;

    await writeCommissionRecord({
      agent: owner.id, level: 2,
      base: tx.amount, rate: rateB, upstream: rateA,
      amount: bCut,
    });
    await writeCommissionRecord({
      agent: A.id, level: 1,
      base: tx.amount, rate: rateA, upstream: null,
      amount: aCut,
    });
  } else {
    // Case 2: 玩家歸屬在 A
    const rateA = await getEffectiveRate(owner.id, tx.paidAt);
    await writeCommissionRecord({
      agent: owner.id, level: 1,
      base: tx.amount, rate: rateA, upstream: null,
      amount: tx.amount * rateA,
    });
  }
}
```

### 結算排程（NestJS）

```typescript
@Injectable()
export class SettlementScheduler {
  @Cron('0 0 * * *') // 每天 00:00
  async checkAndSettle() {
    const today = new Date();
    const settlementDay = await this.settings.get('settlement_day');
    const lastDayOfMonth = getLastDayOfMonth(today);
    const effectiveDay = Math.min(settlementDay, lastDayOfMonth);

    if (today.getDate() !== effectiveDay) return;

    const [periodStart, periodEnd] = this.calcPreviousPeriod(today, settlementDay);
    await this.settlementService.createSettlements(periodStart, periodEnd);
  }
}
```
